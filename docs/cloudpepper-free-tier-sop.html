Cloudpepper Free Tier — Standard Operating Procedure

New instance → server prep → restore a database backup → add the custom module. Tested on Odoo 19 on a DigitalOcean droplet.

Free-tier limits to remember: the Cloudpepper free tier has no Backup-restore UI and no Addons/Git tab. Both database restores and custom-module deploys are done on the server (commands below), not through the dashboard.
Preferred way to run server commands: from PowerShell on your PC via ssh root@<SERVER_IP>. The DigitalOcean web Console is used only for the one-time SSH-key setup, or as a fallback if SSH isn't available.

1. Prerequisites & conventions

  • A Cloudpepper account (free tier) managing an Odoo 19 instance on your own DigitalOcean droplet.
  • Your Odoo backup as an Odoo .zip (contains dump.sql + filestore/ + manifest.json).
  • Your custom module in a public GitHub repo (private needs a token — public is simplest for git pull deploys).

2. Create a new instance

  1. In Cloudpepper, create a new Odoo instance on your DigitalOcean server.
  2. Under Advanced Options, set a unique database name (<DB>) — write it down; it's used everywhere below.
  3. Wait for the instance to show Running. Note its IP (<SERVER_IP>).

3. Get a server shell (PowerShell + SSH — preferred)

Run all server commands from PowerShell on your PC via SSH. You only need the DigitalOcean Console once, to install your key.

One-time — install your SSH key

In PowerShell, create a key (skip if you already have one) and show its public half:

ssh-keygen -t ed25519
Get-Content $env:USERPROFILE\.ssh\id_ed25519.pub

Open the DigitalOcean web Console once (Droplet → Console; if it needs a password use Access → Reset Root Password) and paste your public key in:

mkdir -p ~/.ssh && chmod 700 ~/.ssh
echo "PASTE_YOUR_PUBLIC_KEY_LINE_HERE" >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys

Every time after — connect from PowerShell

ssh root@<SERVER_IP>

Enter your key passphrase; the prompt becomes root@…:~#. Run all the server commands below here. (The DigitalOcean web Console remains a fallback if SSH is ever unavailable.)

Two windows, don't mix them up: PS C:\…> is your PC; root@…:~# is the server (after you SSH in). scp runs on your PC; the restore / module commands run on the server.

4. Add swap (prevents out-of-memory crashes)

Small droplets (1–2 GB) run Odoo right at the memory ceiling. Add a 2 GB swap file once (run on the server):

swapoff /swapfile 2>/dev/null; rm -f /swapfile
fallocate -l 2G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
grep -q '/swapfile' /etc/fstab || echo '/swapfile none swap sw 0 0' >> /etc/fstab
echo 'vm.swappiness=10' > /etc/sysctl.d/99-swappiness.conf
sysctl vm.swappiness=10
free -h

Success: the Swap row in free -h shows 2.0Gi.

Swap stops crashes but lives on disk (slower than RAM). If the server is constantly full, resize the droplet RAM in DigitalOcean (CPU & RAM-only resize, reversible).

5. Restore a database backup

5a. Upload the backup to the server

From PowerShell on your PC (needs the SSH key from step 3B):

scp "C:\path\to\your-backup.zip" root@<SERVER_IP>:/root/backup.zip

Wait for the 100% progress line.

5b. Restore it (run on the server)

This replaces the <DB> database. The site is briefly down during the import.
systemctl stop odona-<DB>
cd /root
apt-get install -y unzip
unzip -o backup.zip
sudo -u postgres psql -c "SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE datname='<DB>' AND pid <> pg_backend_pid();"
sudo -u postgres psql -c "DROP DATABASE IF EXISTS \"<DB>\";"
sudo -u odoo psql -d postgres -c "CREATE DATABASE \"<DB>\" ENCODING 'unicode' LC_COLLATE 'C.UTF-8' LC_CTYPE 'C.UTF-8' TEMPLATE template0;"
cat dump.sql | sudo -u odoo psql -d <DB>
sudo rm -rf /var/lib/odoo/.local/share/Odoo/filestore/<DB>
sudo mv filestore /var/lib/odoo/.local/share/Odoo/filestore/<DB>
sudo chown -R odoo:odoo /var/lib/odoo/.local/share/Odoo/filestore/<DB>
rm -f dump.sql backup.zip manifest.json
systemctl restart odona-<DB>
If systemctl stop odona-<DB> says Unit not found, find the real service name with systemctl list-units | grep -i odo.

6. Add the custom module

Find the instance dir and the addons path (run on the server):

systemctl cat odona-<DB> | grep -i workingdirectory
grep -i addons_path <INSTANCE_DIR>/odoo.conf

Clone your repo into extra-addons/ and add it to the addons path:

apt-get install -y git
git clone <REPO_URL> <INSTANCE_DIR>/extra-addons/my-module
# match ownership to the existing addons so Odoo can read it:
chown -R $(stat -c '%U:%G' <INSTANCE_DIR>/src/addons) <INSTANCE_DIR>/extra-addons/my-module
# append the repo folder (the one CONTAINING the module) to addons_path:
grep -q 'my-module' <INSTANCE_DIR>/odoo.conf || sed -i 's#^addons_path = .*#&, <INSTANCE_DIR>/extra-addons/my-module#' <INSTANCE_DIR>/odoo.conf
grep addons_path <INSTANCE_DIR>/odoo.conf
systemctl restart odona-<DB>
If your repo is private, git clone asks for credentials (GitHub no longer accepts your account password). Either make the repo public, or use a Personal Access Token (Settings → Developer settings → Tokens (classic), scope repo) as the password.

Then register it in Odoo (browser):

  1. Settings → bottom → Activate developer mode.
  2. Apps → ⋮ → Update Apps List.
  3. In the Apps search bar, remove the default "Apps" filter chip (it hides non-application modules).
  4. Search your module → Activate / Install.
Tip: if the module doesn't appear, the code isn't on the addons path — re-check the addons_path line points at the folder containing the module's __manifest__.py.

7. Ongoing module updates

After you push new commits to GitHub: from PowerShell, ssh root@<SERVER_IP>, then:

cd <INSTANCE_DIR>/extra-addons/my-module && git pull && systemctl restart odona-<DB>

Then in Odoo: Apps → (your module) → Upgrade. For pure JS/XML asset changes, also hard-refresh the browser (clear the POS cache if it looks stale).

First pull after a fresh clone may fail with "dubious ownership" (the clone is owned by the odoo user but git runs as root). Whitelist it once, then re-run:
git config --global --add safe.directory <INSTANCE_DIR>/extra-addons/my-module
If a pull leaves files root-owned, restore ownership: chown -R odoo:odoo <INSTANCE_DIR>/extra-addons/my-module.
