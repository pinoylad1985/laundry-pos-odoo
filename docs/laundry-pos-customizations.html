Laundry POS — Customizations Reference

A complete inventory of everything the laundry_pos module changes in Odoo 19 POS — models, behaviors, and the design reasons behind them. Generated from the source on branch main.

This is the technical/feature reference. For end-user click-through usage see laundry-pos-module-guide.html; for deployment (self-hosted Docker Compose, production + staging) see CLAUDE.md in the repo root.
Important change vs. older docs: the original "Laundry Services tab" product-tagging and service-type product filtering have been removed. The module now recognises 4 fixed service products by NAME (Wash-Dry-Fold, Dry/Wet Clean, Shoe Clean, Press) and the New Order modal's service pills add those products to the order. There is no product_template.py and no product views in the current module.

2. Backend models & fields

laundry.service.type models/laundry_service_type.py

Simple catalog model (name, code, sequence, active). Seeded with 5 types via data/laundry_service_type_data.xml (noupdate="1"): Drop-off, Drop-off & Delivery, Pickup & Delivery, Locker, Self-service. (The POS frontend currently uses its own hardcoded list of the same 5 types.)

pos.order extra fields models/pos_order.py

FieldTypeNotes
laundry_customer_typeSelectionnew / returning
laundry_service_typeSelectiondropoff / dropoff_delivery / pickup_delivery / locker / self_service
laundry_turnaroundSelectionexpress / regular
laundry_servicesCharcomma-joined service names
laundry_claim_datetimeDatetimeset from modal submit
laundry_pickup_datetimeDatetimeset from modal submit
laundry_delivery_datetimeDatetimeset from modal submit
laundry_due_datetimeDatetime, computed+stored"Cashier Due" = delivery datetime, else claim datetime
laundry_customer_phoneChar, computed+storedfrom partner_id.phone
laundry_customer_addressChar, computed+storedstreet + street2
Why computed-stored, not related: a stored related field is writable and would push values back to res.partner — POS could wipe the customer's phone on order sync. A computed field has no inverse, so it stays one-way (partner → order) and any stale value POS sends is corrected on the next recompute. Also: no _load_pos_data_fields override on pos.order — its default returns [] (read ALL fields), so the selection fields load automatically; overriding it would break the order load.

res.partner.pos_tag_names models/res_partner.py

Computed comma-joined category_id names, added to _load_pos_data_fields.

Why: the receipt needs customer tags without loading the res.partner.category model. Relational loading only pulls categories referenced by pre-loaded partners, so tags on customers fetched on-demand (cashier search) would otherwise be missing. A precomputed string on the partner is always present. The same on-demand concern drives the multi-word server search (§5).

3. Service products (name matching) utils/laundry_instructions.js, laundry_products.js

Four service products are recognised by name (no hardcoded IDs):

CodeLabelName match
wdfWash-Dry-Fold"wash-dry-fold"
dwcDry/Wet Clean"dry/wet clean"
shoeShoe Clean"shoe clean"
pressPress"press"

dwc and shoe are "long-turnaround" services (affect schedule thresholds). Matching uses a leading word boundary so "press" never matches "(express)". Lookups go through live pos.models (active + available_in_pos), so price/attributes/variants always reflect the real POS product.

4. New Order modal & turnaround logic new_order_modal.js, product_screen_patch.js, pos_store.js

The setup modal is opened by the navbar New Order button (or the banner's Complete button) for the current order — see §5. It is not auto-opened.

The 4 steps

  1. Customer — New / Returning; live multi-word partner search (name, phone, mobile, street, city…) with select or "edit". Press Enter to also search the server.
  2. Services — pressing a pill calls addLineToCurrentOrder, adding the real product as an unconfigured line. Pills highlight while ≥1 line of that product is on the order.
  3. Service Type — the 5 types; changing it resets the schedule.
  4. Schedule — quick date pills (Today / Tomorrow / +2 days) + AM/PM hour pills. Closed hours: 4–5 AM for all selectors; pickup & delivery also block 2–3 AM.
Confirm requirements: Continue stays disabled until every required field is set — including a selected customer when the type is "Returning" (New customers need none), and a service line unless the service type is Self-service (which needs none). This is enforced whether filling out the first time or after a "Change". The chosen customer is shared with the order (= the Customer Control Button, whichever is set first), pre-selected on re-open with an Unselect button to change it.

Turnaround (TAT) — auto-decided by the schedule, not the cashier

Express thresholdMinimum lead time
Standard (WDF / Press)< 24h new, < 18h returning6 hrs
Long (Dry/Wet Clean, Shoe Clean)< 72h new, < 48h returning24 hrs

The modal computes hours between now (or pickup) and claim/delivery, then labels the order Express (below threshold) or Regular. Below the minimum lead time it shows a validation error and blocks confirm. Skip for now still saves whatever was selected.

5. Navbar hub buttons, mutual exclusivity, Settle & Order List navbar_patch.js/.xml, product_screen_patch.js, settle_modal/, ticket_screen_patch.*

Three buttons live in the navbar (desktop = text, mobile = icons): New Order, Settle Order, Order List. A blank order has no chosen purpose — there is no setup banner for it, just a locked product grid reading "Tap New Order or Settle Order above to begin". Clicking a button decides the order's purpose.

  • New Order → opens the New Order setup modal for the current order. It deliberately does not create or navigate to a new order (that navigation fails under the POS service worker); use the native ＋ to start the next order. The navbar fires a laundry-action DOM event handled by ProductScreen._runLaundryAction.
  • Settle Order → opens the Settle modal (settle_modal/): a customer search where each row surfaces that customer's real settlement action — Settle orders / Settle invoices / Settle due amount / Deposit money — derived from getPartnerCredit and run through the stock pos_settle_due flows. A customer already on the order (Control Button) pre-fills it, shown with Unselect + the applicable settle buttons together.
  • Order List → navigates to the Orders screen (TicketScreen); ticket_screen_patch adds a customer-search bar above the core "Search Orders" bar (the two are mutually exclusive — using one greys out the other).

Mutual exclusivity — an order is either a sale or a settlement

Clicking New Order or Settle Order sets order._laundryPurpose ('sell' / 'settle') immediately. The navbar getters laundryActiveSell / laundryActiveSettle then highlight the chosen button (solid primary) and disable the opposite one. The flag is restored after a reload from the stored status / settle lines. To switch an order's type, start a fresh order with ＋.

Why the buttons need z-index: the core pos-leftheader contains an invisible position-relative w-100 layer. By CSS painting rules a positioned element paints over static ones, so that layer sat on top of the leftmost button (New Order) and swallowed its clicks. The buttons are lifted with position:relative; z-index — don't remove it.

Multi-word customer search (used everywhere)

utils/partner_search.js provides partnerMatchesQuery (every typed word must match some field — name, parent name, phone, mobile, street, city, zip, VAT, email, barcode…) and buildPartnerSearchDomain for an Enter-to-search server lookup (POS only pre-loads a subset of customers, so a name like "Val" may not be loaded yet). It is used by the New Order modal, Settle modal, Order List bar, and the core Customer picker. partner_block_patch redirects the core "Settle / Deposit" partner-line menu items into the Settle modal so there's one settle path.

Order List is independent of the Customer Control Button. PosStore.getDefaultSearchDetails is overridden to always return a blank search; core otherwise seeds the Order List's order search with the current order's partner name, coupling the two.

6. Setup banner, grid lock & persistence product_screen_patch.js/.xml, laundry_storage.js, pos_store.js

The strip above the cart reflects the current order's setup state:

  • Blank / idle order — no banner. The product grid is locked with an overlay ("Tap New Order or Settle Order above to begin"); the cart stays usable so a settlement can use it.
  • Submitted — a green summary bar (customer type, services, service type, TAT, schedule) with a Change button to re-open the modal; the grid unlocks.
  • Skipped / incomplete — an amber bar with a Complete button to finish setup.
  • The Customer button is blocked until setup is done — PosStore.selectPartner returns false and dispatches a laundry-flash-needed event to flash the banner.
  • Details persist in localStorage keyed by order UUID (rotating out the oldest beyond 100 entries), and are purged on order delete/cancel so a reused UUID can't resurrect stale data.
Why localStorage: the laundry_* JS fields are not synced to the server, so they vanish on POS reload / model rebuild. The store rehydrates them (and the modal pre-fills from it on "Change").

7. Cart lines: merge, quantity, ordering pos_store.js, pos_order_line_patch.js, order_display_patch.js

  • Never merge lines — tryMergeOrderline forces merge=false, so every pill press / tap is its own line. Quantity is adjusted via the numpad.
  • Quantity rules (PosOrderline.setQuantity): Dry/Wet Clean & Shoe Clean are locked to 1 (0 still allowed so the line can be deleted); Wash-Dry-Fold is per-KG (editable); Press may exceed 1.
  • Fixed display order (OrderDisplay.comboSortedLines): WDF → Press → Dry/Wet Clean → Shoe Clean → everything else. Used by both the cart and the receipt so they always match.

8. Configurator behavior & crash guard order_summary_patch.js, product_configurator_popup_patch.js

  • Tap a laundry line → configurator. Tapping opens a fresh configurator (productTemplate only); after confirming, the line is swapped for the configured one (no-merge keeps it separate), quantity preserved, and it's selected so the numpad can adjust qty/weight.
  • Pre-fill on re-open — the configurator restores the line's existing attribute selection rather than a blank form.
  • Turnaround is locked to the order's TAT — the configurator pre-selects the matching Express/Regular value and the template greys it out; a schedule change re-applies TAT to already-configured lines.
  • Crash guard — initAttributes() pre-creates a state entry for every value's attribute_id before calling core.
Why the guard / fresh configurator: core's onOrderlineLongPress assumes a fully-configured line and crashes on ours, so we always open a fresh one. Separately, core seeds state.attributes by each attribute line's id but then sets .selected keyed by each value's attribute_id; a product with a value whose attribute isn't a line key (malformed / mis-migrated, e.g. after a DB restore) throws "Cannot set properties of undefined". Pre-seeding every key prevents the whole popup from crashing.

9. Payment gates pos_store.js

Both Pay buttons funnel through PosStore.pay(), which blocks with an AlertDialog when:

  • any laundry line still has unselected options (lists the offending product names); or
  • a Wash-Dry-Fold minimum weight is unmet — at least 6KG for a single WDF line, or 4KG on each line when there is more than one WDF line.

10. Receipt customization order_receipt_patch.js, receipt_header_patch.js, order_line_patch.js, laundry_receipt.scss

Multi-copy thermal receipt

computeLaundryCopies(order) builds, for a laundry order:

  • one "i/n TRANSACTION COPY" per main-service line (that line boxed), ordered WDF → Press → DWC → Shoe;
  • a SHOP COPY (always);
  • a CUSTOMER COPY only for Drop-off, Drop-off & Delivery, Self-service.

Non-laundry orders → a single normal receipt. Each copy is printed as its own print job (PosStore.printReceipt loops with setPrintOnlyCopy) so the thermal printer cuts between copies.

Header meta & per-line formatting

  • Header block (receipt_header_patch.js): customer type, services, service type, TAT, pickup, and claim/delivery datetimes (12-hour). Resolves from order JS fields or localStorage for reprints.
  • Line formatting (order_line_patch.js): per-line attributes as name/value; "Turnaround (Press)" / "Turnaround Time" normalised to "Turnaround" and dropped from the printed line (shown once in the header); the duplicated "(Variant)" suffix stripped from the product name; WDF/Press get write-in count lines (Sorting / Folding / Press counts).

12. Maintenance: Demangle & Split Customer Names data/demangle_server_action.xml

A one-time server action (Contacts list → Action menu), not part of the POS runtime — it just rides in this module as the deploy vehicle. For each selected contact it keeps the part before the first " | " (undoing an "| ADDRESS | PHONE | TAGS" mangling), then splits the recovered name by "/" and "-" into name / street / street2.

Guard: it refuses to run while an active "Update Customer Name" automation exists (Settings → Technical → Automation Rules) — otherwise names re-mangle on save. Delete/archive that automation first.

13. Key design decisions (why)

  • addNewOrder stays synchronous — async breaks order-UUID routing.
  • New Order opens the modal for the current order, never navigates — creating/navigating to a new order fails under the POS service worker; the native ＋ starts the next order.
  • Mutual exclusivity via _laundryPurpose — set on click, so Settle/New Order disable each other immediately; restored after reload from stored status / settle lines.
  • Navbar buttons lifted with z-index — the core left-header's invisible positioned layer otherwise paints over and eats clicks on the leftmost button.
  • Order List decoupled from the order's customer — getDefaultSearchDetails returns a blank search so the two selections stay independent.
  • No _load_pos_data_fields on pos.order — default [] loads all fields; overriding would drop lines/partner/amounts.
  • Computed (not related) order fields — keep partner data one-way so POS sync can't overwrite the customer record.
  • Name matching, not product IDs/tags — survives DB restores and re-seeding; no per-product configuration needed.
  • localStorage for laundry meta — the JS-only fields aren't server-synced, so they must be rehydrated across reloads.
  • Turnaround driven by schedule — the TAT is computed and locked, not a free cashier choice, so receipts and lines stay consistent.
  • Multi-word + server partner search — POS pre-loads only a subset of customers, so search must also query the server (Enter) to find un-loaded contacts.
